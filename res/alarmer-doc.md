# Alarmer — документация

Бот слежки за Telegram-каналами с фильтрацией и пересылкой в твои каналы. Работают два процесса в паре: **userbot** (читает каналы как пользователь) и **bot** (принимает события, фильтрует и публикует через Bot API).

## Как это работает

```
[Telegram-канал-источник]
        │  (юзер-аккаунт читает)
        ▼
   userbot (Pyrogram)
        │  POST /bot/ub
        ▼
   bot (Bot API + FastAPI)
        │  фильтр по триггерам / стоп-словам / cut
        │  send_message(@твой_канал)
        ▼
[Твой канал-получатель]
```

- **userbot** заходит как настоящий пользователь Telegram (по номеру телефона). Раз в `POLL_INTERVAL_SECONDS` (по умолчанию 5 сек) подтягивает свежую историю из каждого источника, фильтрует свежие сообщения (моложе 40 минут, не виденные ранее) и пушит их боту по HTTP.
- **bot** — официальный Telegram-бот (по `BOT_TOKEN`). Принимает данные от userbot'а, прогоняет через фильтры из настройки ресурса, и публикует результат в канал-получатель. Также обслуживает админ-интерфейс (`/start`).

**Зачем два процесса.** Bot API не умеет читать произвольные каналы — только те, где бот сам админ. Userbot читает любой канал, на который подписан пользователь. Так мы получаем доступ ко всему, что доступно глазами.

**Источник → получатель = «ресурс».** Один источник можно подключить к разным получателям с разными фильтрами; в одном получателе может сидеть несколько ресурсов с разных источников.

## Установка и запуск

### Что нужно

- Linux-сервер с docker и docker-compose
- Доменное имя с DNS на сервер (для Telegram webhook'а)
- Telegram Bot Token — у [@BotFather](https://t.me/BotFather)
- Telegram API ID + API Hash — на [my.telegram.org](https://my.telegram.org/apps)
- Номер телефона пользователя, под которым будет работать userbot (обычный аккаунт; полезно завести отдельный)

### Первый запуск

1. Клонируешь репозиторий и заполняешь `.env` (см. ниже).
2. Активируешь userbot-сессию (одноразово, интерактивно):
   ```bash
   make init
   ```
   Бот напишет тебе в Telegram, попросит подтвердить номер, прислать SMS-код, ввести 2FA-пароль если есть. Появится файл `.sessions/alarmer.session` — это сохранённая авторизация userbot'а.
3. Запускаешь оба процесса:
   ```bash
   docker compose up -d --build
   docker compose logs -f
   ```
4. Настраиваешь reverse-proxy (см. раздел Caddy ниже).
5. Открываешь бот в Telegram (`/start`), дальше всё через инлайн-кнопки.

### Бэкап

Достаточно сохранять:
- `.sessions/` — авторизация userbot'а (без неё придётся проходить `make init` заново)
- `${DB_PATH}` (по умолчанию `alarmer.db`) — все настроенные каналы и ресурсы

## Конфигурация (.env)

Пример заполненного `.env`:

```dotenv
# ── Бот (Bot API) ────────────────────────────────────────
BOT_TOKEN=1234567890:AAFakeTokenExampleXXXXXXXXXXXXXXXX
BOT_DROP_UPDATES=no
BOT_API_PREFIX=/bot

ADMINS=123456789,987654321              # Telegram user-id админов (через запятую)

DB_PATH=alarmer.db

# ── Webhook ──────────────────────────────────────────────
WEBHOOK_URL=https://alarmer.example.com  # домен, на котором стоит реверс-прокси
WEBHOOK_SECRET=<openssl rand -hex 32>    # секрет, общий для всех HTTP-эндпоинтов
WEBHOOK_HOST=0.0.0.0
WEBHOOK_PORT=8000

# ── Userbot (Telegram user account) ──────────────────────
UB_API_ID=12345678
UB_API_HASH=0123456789abcdef0123456789abcdef
UB_PHONE_NUMBER=79001234567
UB_TWOSTEP_PASSWORD=mypassword           # если в аккаунте 2FA; иначе оставь пустым

UB_APP_VERSION="Piggygram 0.7.4"
UB_DEVICE_MODEL=SM-S928B
UB_SYSTEM_VERSION="Android 16"
UB_LANG_PACK=android
UB_LANG_CODE=en
UB_SESSION_PATH=.sessions

# /userbot/call HTTP API
UB_CALL_HOST=0.0.0.0
UB_CALL_PORT=8001
UB_CALL_URL=http://userbot:8001          # имя сервиса в docker-compose
UB_CALL_TIMEOUT_SECONDS=30
UB_API_PREFIX=/userbot

# ── Поллер ───────────────────────────────────────────────
POLL_INTERVAL_SECONDS=5
POLL_HISTORY_LIMIT=20
POLL_REQUEST_TIMEOUT_SECONDS=10
POLL_MAX_CONCURRENCY=10
```

### Что значит каждое поле

| Переменная | Описание |
|---|---|
| `BOT_TOKEN` | Токен бота от @BotFather |
| `ADMINS` | User-id'ы тех, кому позволено управлять ботом. Все остальные сообщения молча игнорируются |
| `WEBHOOK_URL` | Внешний URL, на котором Telegram достучится до бота. Без хвостового слэша и пути |
| `WEBHOOK_SECRET` | Общий секрет. Им проверяется `X-Telegram-Bot-Api-Secret-Token` на всех POST-эндпоинтах (бот→userbot, userbot→бот, Telegram→бот) |
| `BOT_API_PREFIX` / `UB_API_PREFIX` | Префиксы для HTTP-эндпоинтов. Telegram'у бот сам скажет `WEBHOOK_URL + BOT_API_PREFIX/webhook` |
| `UB_API_ID`, `UB_API_HASH` | С my.telegram.org/apps |
| `UB_PHONE_NUMBER` | Номер для авторизации userbot'а |
| `UB_TWOSTEP_PASSWORD` | 2FA-пароль аккаунта. Можно опустить, если не выставлен |
| `POLL_INTERVAL_SECONDS` | Раз в сколько секунд тикать. 5 сек = баланс между скоростью и нагрузкой |
| `POLL_HISTORY_LIMIT` | Сколько последних сообщений тянуть из каждого канала за тик. 20 — норм |
| `POLL_MAX_CONCURRENCY` | Сколько источников одновременно опрашивать. 10 — норм |

## Reverse-proxy (Caddy)

Только **одна** ручка должна быть доступна публично — `POST /bot/webhook` (это куда стучится Telegram). Всё остальное — внутреннее.

```caddyfile
alarmer.example.com {
    # Только Telegram. Остальное закрыто.
    handle /bot/webhook {
        reverse_proxy localhost:8000
    }

    # Опционально — для uptime-чекера снаружи
    handle /bot/ping {
        reverse_proxy localhost:8000
    }

    handle {
        respond 404
    }

    encode zstd gzip
}
```

Каддишка автоматически выпустит TLS-сертификат для домена.

### Проверка после деплоя

```bash
# 1) Локально бот живой
curl -i http://localhost:8000/bot/ping
#    HTTP/1.1 200 OK  → Pong!

# 2) Снаружи проксируется
curl -i https://alarmer.example.com/bot/ping
#    HTTP/2 200       → Pong!

# 3) Telegram видит свой webhook
curl "https://api.telegram.org/bot$BOT_TOKEN/getWebhookInfo"
#    url: https://alarmer.example.com/bot/webhook
#    pending_update_count: 0
```

## Использование бота

Открываешь бота, шлёшь `/start`. Главное меню — две кнопки:

- **Добавить ресурс** — мастер настройки нового канала-получателя + источника.
- **Мои каналы** — список настроенных каналов-получателей с краткой статистикой.

### Добавление ресурса (мастер)

Бот ведёт за руку через 6 шагов. На любом шаге можно прислать `/cancel` (или `cancel`, или `отмена`).

1. **Канал-получатель.** Куда слать уведомления. Принимает:
   - Числовой ID: `-1001234567890`
   - Юзернейм: `@my_channel`
   - Ссылку: `https://t.me/my_channel`
   - Приватный инвайт: `https://t.me/+abc...` или `https://t.me/joinchat/abc...`
2. **Источник.** За кем следим. Те же 4 формата.

   Для **приватных каналов** (и в канале-получателе, и в источнике) userbot автоматически попробует вступить через инвайт. Для публичных — просто резолвит ID через `getChat`. При этом покажется промежуточное сообщение «Резолвлю канал…» / «Вступаю в канал…».

   ⚠️ **Бот** должен быть админом в канале-получателе (Telegram не даёт ботам автоматически вступать в каналы). Это ты добавляешь руками отдельно.

3. **Режим.** Выбираешь кнопкой:
   - **Как есть** — пересылаем исходный текст (с фильтром по ключевым словам). Это RAW-режим.
   - **Свой шаблон** — на определённое сообщение шлём свой текст. Это EDITABLE-режим.

4. **Триггеры / шаблоны.** Зависит от режима — см. [следующую главу](#типы-ресурсов-raw-и-editable).

5. **Убирать из текста.** Подстроки, которые вырежем из пересылаемого сообщения. Каждая с новой строки или через запятую. `-` — пропустить шаг.

6. **Стоп-слова.** Если в сообщении встретилось — оно не пересылается, даже если триггер сработал. Каждое с новой строки или через запятую. `-` — пропустить.

После создания открывается **карточка ресурса** — со всеми настройками и кнопками для их правки.

### Карточка ресурса

```
📌 @источник  (🟢 активен)
Слать в: ИМЯ ПОЛУЧАТЕЛЯ
Режим: Как есть · регулярки: выкл · эмодзи: оставляем

Ключевые слова
[первые 20 ключевых слов в pre-блоке]
… ещё N. Жми «Скачать .txt», чтобы увидеть всё.

Стоп-слова: word1, word2
Убирать из текста: phrase1
Подпись: —
```

Кнопки:
- **Ключевые слова / Шаблоны** — изменить триггеры. Можно прислать `.txt`-файл вместо текста (до 5 МБ).
- **Подпись** — текст, который приклеивается в конец каждой пересылки. Поддерживается `{username}` (заменится на @имя источника) и форматирование Telegram'а (жирный, курсив, ссылки).
- **Стоп-слова**, **Убирать из текста** — то же что в мастере, только правишь по месту.
- **Эмодзи: вырезать / оставлять** — выкосить ли все эмодзи из текста перед пересылкой.
- **Регулярки: вкл / выкл** — переключатель + меню флагов `i m s x` (см. ниже).
- **На паузу / Включить** — выключить отслеживание, не удаляя ресурс.
- **Скачать .txt** — скачать триггеры/шаблоны файлом для редактирования.
- **Удалить** — удаление с подтверждением.

### Удаление ресурса

Жмёшь «Удалить», подтверждаешь. Если в канале-получателе больше нет других ресурсов, он сам удаляется из списка.

## Типы ресурсов: RAW и EDITABLE

### Синтаксис триггеров (общий для обоих режимов)

Триггер — это **выражение из блоков**. Сообщение матчится, если хотя бы один блок целиком попал в текст.

| Оператор | Смысл | Пример |
|---|---|---|
| `\|` | OR между блоками (top-level) | `тревога\|опасность` — любое из двух |
| `+` | AND внутри блока — все части должны встретиться в тексте | `отбой+опасности` — оба слова |
| `(…)` | Группирующие скобки, по желанию, для читаемости | `(отбой+опасности)\|(отмена+тревоги)` |
| `\+`, `\|`, `\(` | Литералы (актуально в regex-режиме) | `\+` = плюс как символ |

Порядок частей в AND-блоке не важен — `отбой+опасности` сматчит и «отбой воздушной опасности», и «опасности больше нет, отбой».

Регистр по умолчанию не учитывается (`casefold`-сравнение).

В **raw-режиме** блоки разделяются переводом строки, запятой или `|`. В **editable-режиме** — только `|` (одна строка = один шаблон).

#### Примеры выражений

| Что хотим поймать | Выражение |
|---|---|
| Любую тревогу | `тревога` |
| Тревогу или опасность | `тревога\|опасность` |
| Только сообщения про отбой именно тревоги | `отбой+тревог` |
| Отбой тревоги ИЛИ отмена опасности | `(отбой+тревог)\|(отмена+опасност)` |
| БПЛА с числом | `БПЛА+ед` |
| Любое сообщение | оставить поле пустым (`-` в мастере) |

### RAW («Как есть»)

Бот ждёт от источника сообщение, проверяет триггер. Если совпало — **пересылает оригинальный текст** (с сохранением форматирования), вырезав `cut`-фразы и применив подпись.

**Формат ключевых слов:** один блок на строку (или через запятую/`|`). Внутри блока — `+` для AND-частей.

```
тревога
БПЛА+ед
(отбой+опасности)|(отмена+тревоги)
ракетная опасность
```

Что здесь происходит:
- строка 1: матчит любое сообщение со словом «тревога»;
- строка 2: только если в сообщении есть и «БПЛА», и «ед» (например, «обнаружено 3 ед. БПЛА»);
- строка 3: два альтернативных AND-блока в одной строке;
- строка 4: одиночная подстрока.

**Пустой список** = ловим **все** сообщения из канала. Полезно для тупой ретрансляции мониторинговых лент. В мастере введи `-`.

### EDITABLE («Свой шаблон»)

Бот ждёт от источника сообщение, прогоняет каждый шаблон. Если левая часть какого-то шаблона матчится — **посылает правую часть** вместо оригинала. Это для подмены формулировок: пришло сухое «Воздушная тревога», ты публикуешь свой текст с эмодзи и инструкцией.

**Формат шаблонов:**

```
выражение => текст для пересылки
другое выражение => другой текст
```

Каждая строка — один шаблон. Слева — выражение в синтаксисе блоков (см. выше). Справа — готовый текст (поддерживает форматирование Telegram'а, перенос строки через `<br>`).

**Пример:**

```
тревога|внимание|опасность => 🚨 <b>Внимание!</b> В регионе объявлена тревога.<br>Не паникуйте, следуйте указаниям властей.
(отбой+тревог)|(отмена+опасност) => ✅ <b>Отбой.</b> Угроза миновала.
БПЛА+ед => ⚠️ <b>БПЛА.</b> Сохраняйте бдительность.
```

Шаблоны проверяются сверху вниз — срабатывает **первый** подошедший. Если ни один не сматчился — сообщение не пересылается.

#### Порядок шаблонов имеет значение

Чем уже выражение — тем выше его ставь. Иначе общее перехватит частное:

```
# Плохо: «отбой» сработает раньше, чем шаблон про учения
тревога|отбой => общая формулировка
учения+отбой => 📚 Учения завершены.

# Хорошо: сначала специфика, потом общее
учения+отбой => 📚 Учения завершены.
тревога|отбой => общая формулировка
```

### Cut, blacklist, footer — общие для обоих режимов

| Поле | Что делает |
|---|---|
| **Стоп-слова (blacklist)** | Если в сообщении есть — оно **не** пересылается, даже если триггер сработал |
| **Убирать из текста (cut)** | Подстроки, которые молча удаляются из итогового текста |
| **Подпись (footer)** | Текст, приклеиваемый в конец каждой пересылки. `{username}` → @имя_источника |
| **Эмодзи** | Можно выкосить все эмодзи из итогового текста |

### Регулярки

В карточке ресурса есть «Регулярки: выкл/вкл». Если включено — каждая AND-часть блока компилируется как Python regex; матч идёт через `re.search`. Операторы `|`, `+`, `(`, `)` на уровне синтаксиса триггеров остаются мета-символами **для нашего парсера** — чтобы получить их как литералы regex, экранируй: `\+`, `\|`, `\(`, `\)`.

Пример: `\bопас\w*+бпла` — два regex'а в AND-блоке: «опас…» и «бпла». Чтобы поймать regex `a+` (один или более `a`), пиши `a\+`.

Флаги (тоже из меню):
- **i** — без учёта регистра
- **m** — `^` и `$` срабатывают на каждой строке
- **s** — точка `.` ловит перенос строки
- **x** — игнорировать пробелы и комментарии в шаблоне

Регулярки, которые не скомпилировались, тихо пропускаются (в логи WARNING, бот не падает). Если хоть одна AND-часть блока не скомпилировалась — весь блок выкидывается из выражения.

## Команда /loadcfg — массовая загрузка

Когда нужно настроить десятки ресурсов сразу — `/start` неудобен. Шлёшь боту `/loadcfg`, в ответ — присылаешь текст или `.txt`-файл с конфигом.

### Формат

```
# Комментарии и пустые строки игнорируются.

# Каждая группа начинается со строки '> @target_or_id_or_link'.
> @yaroslavll_live

# Внутри группы — источники. Заголовок: '@source MODE', где MODE = raw или editable.
@evraevmikhail editable
тревога|внимание => 🚨 Ярославль: внимание

# В raw-режиме — ключевые слова через запятую/переносы
@radarrussiia raw
Ярославль, Ярославская область, Рыбинск, Тутаев

# Следующая группа
> @live_piterskiy

@drozdenko_au_lo editable
тревога => 🚨 Питер: тревога

@radarrussiia raw
Санкт-Петербург, Питер, СПб, Ленобласть
```

**Правила:**

- Target и source могут быть в любом из 4 форматов (ID, `@user`, `t.me/...`, `+invite`).
- В editable можно ничего не писать после заголовка — это валидная заглушка, дозаполнишь триггеры через UI потом.
- В raw пустой список или `*` / `-` = «ловим всё».
- Дубли (тот же target+source) пропускаются с пометкой `⏭️ уже добавлено`.

Бот покажет прогресс по мере обработки, в конце — сводку «добавлено X, проблем Y» с детализацией ошибок (например, «канал не найден» если userbot не подписан или юзернейм опечатан).

### Готовый конфиг для региональных лент

Лежит в `.resources/regions.txt` рядом с этой документацией. Если рабочий — пришли его боту через `/loadcfg`.

## Расширенные настройки ресурса

### Подпись с `{username}`

Например, для footer'а:
```
<br><i>Источник: {username}</i>
```

В пересылке станет:
```
... сообщение ...

Источник: @radarrussiia
```

### Cut с многострочным текстом

Если хочешь вырезать многострочный блок (например, рекламную плашку каждого сообщения) — можно прислать его одним куском в `.txt`. Или просто прописать в один линию первого слова цепочки — cut работает по точному совпадению подстрок.

### Регулярки

Включи галку «Регулярки» в карточке. Теперь триггеры — это re-паттерны.

Пример: ловить только конкретное число дронов:
```
БПЛА.{0,10}[0-9]+\s+ед
```

Или ловить упоминание города по любым склонениям одним паттерном:
```
\b[Пп]ермск\w*\b
```

## Сетевая раскладка (для devops)

```
Telegram ──HTTPS──▶ bot     :8000  /bot/webhook    ← публично через caddy
                            :8000  /bot/ub         ← от userbot, через docker network
                            :8000  /bot/ping       ← health

       bot ──HTTP──▶ userbot :8001  /userbot/call  ← резолв/вступление в чаты
                            :8001  /userbot/ping   ← health
```

- Бот и userbot — два контейнера в одном `docker-compose`. Общаются по именам сервисов (`http://alarmer:8000`, `http://userbot:8001`).
- Все POST-эндпоинты авторизованы заголовком `X-Telegram-Bot-Api-Secret-Token` (значение = `WEBHOOK_SECRET`). Без правильного секрета — 403.
- Наружу нужно отдать только `/bot/webhook` (Telegram сам себе DNS). Остальное лучше прятать в reverse-proxy за `404`, хотя секрет защищает в любом случае.

Префиксы (`BOT_API_PREFIX`, `UB_API_PREFIX`) — конфигурируемые, по умолчанию `/bot` и `/userbot`. Подпути внутри них (`/webhook`, `/ub`, `/ping`, `/call`) — хардкод.

## Решение типовых проблем

### Бот не отвечает на /start

1. `docker compose logs -f alarmer` — нет ошибок при старте?
2. Твой user-id внесён в `ADMINS`? Узнать id — у `@userinfobot`.
3. `curl https://alarmer.example.com/bot/ping` — отвечает? Если нет — проблема в caddy/DNS/TLS.
4. `curl "https://api.telegram.org/bot$BOT_TOKEN/getWebhookInfo"` — `last_error_message` пуст? Если там «SSL handshake failed» / «Connection refused» — проверь caddy. Если webhook URL пустой — бот не успел его поставить (рестартни alarmer).

### userbot не подключается, висит на «Connecting…»

1. `make init` ещё не делали? Запусти, пройди авторизацию.
2. Session-файл в `.sessions/` есть? Если потёрли — нужно `make init` заново.
3. `UB_API_ID` / `UB_API_HASH` верные? Регистр и пробелы важны.

### `Waiting for N seconds (required by "messages.GetHistory")`

Telegram прислал FloodWait — слишком частые запросы к этому каналу. Pyrogram сам подождёт и повторит. Если повторяется массово:
- Подняли `POLL_HISTORY_LIMIT`? Опусти обратно к 20.
- Опросов слишком много в секунду? Подними `POLL_INTERVAL_SECONDS`.
- Используешь `chat_id` где должен быть `@username`? Юзернейм надёжнее, его поллер и так предпочитает.

### `CHANNEL_PRIVATE` в логах

Юзербот не имеет доступа к источнику. Варианты:
- Канал приватный, юзербот не вступил — пришли инвайт-ссылку через мастер, он вступит.
- Канал публичный, но юзернейм опечатан в конфиге.
- Канал был публичным, потом перевели в приватный — нужно вступить.

### Бот не пересылает сообщения, хотя в карточке всё настроено

1. Ресурс не на паузе? В карточке должно быть «🟢 активен».
2. Триггер реально совпадает? В RAW-режиме без регулярок — это case-insensitive substring match. В EDITABLE — то же, но матчится список `ключ1|ключ2`.
3. Бот — админ в канале-получателе? Без этого Bot API не даст постить.
4. Не сработал blacklist? Если в сообщении встретилось стоп-слово — оно молча отбрасывается.
5. Логи: `docker compose logs alarmer -f` → ищи строку с `forwarded` или ошибки.

### Хочу полностью переавторизовать userbot

```bash
rm -rf .sessions/
make init
```
(контейнеры можно не останавливать — userbot прочитает новый session-файл при следующем перезапуске)

### Удалить всё и начать с нуля

```bash
docker compose down
rm -f alarmer.db
rm -rf .sessions/
make init
docker compose up -d --build
```

## Поддержка

Логи здесь:
```bash
docker compose logs -f alarmer        # бот
docker compose logs -f alarmer-userbot # юзербот
```

Уровень логов настраивается через `DEBUG=yes` в `.env` (включит DEBUG-уровень, покажет pyrogram внутрянку — много шума).

Файл `alarmer.log` в корне репозитория — то же самое, что в stdout, но без цвета и с временем. Удобно для grep'а постфактум.
